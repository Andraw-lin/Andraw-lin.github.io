# Breaking Changes

在Vue 3.0中，除了一些新的必要的核心特性外，还在Vue 2.x的部分逻辑处理上也进行了一些重大更改（Breaking Changes）。下面我们就来简单总结下🤔

# Global API

Vue 2.x曾经提供给我们可以定义全局的一些组件、指令、配置等。如下：

// 全局组件
Vue.component('my-component', {
  // ...
})

// 全局指令
Vue.directive('focus', {
  // ...
})

上述的代码，对于开发者来说的确很好理解，而且在使用上也没啥难度。但是，当你稍微细心思考一番，你就会发觉出一些问题存在。

首先，Vue 2.x使用的是new Vue()形式来创建页面实例，如果像上述这样来定义一些全局组件或指令，这样一来，对于所有使用new Vue()所创建的页面实例都会被共用全局的配置了。这会产生什么问题？

  • 由于共用相同的配置，就会很容易导致一个页面实例对于全局配置的更改而影响了其他页面实例，也就是产生副作用。
  • 每个页面实例很难做到单独的全局配置。

为了解决上面的疑难杂症，Vue 3.0为我们提供了新的API。

其中，不再采取new Vue()形式来创建页面实例，均采取createApp创建页面实例。

import { createApp } from 'vue'
const app = createApp({})

对于一些常见的全局配置API更改如下：

Vue 2.x Vue 3.0
Vue.config app.config
Vue.component app.component
Vue.directive app.directive
Vue.mixin app.mixin
Vue.use app.use

好明显，通过createApp创建的是一个单独的页面实例，并且可以定义属于它自己的全局配置。

那么问题又来了，既然createApp创建是一个单独的页面实例，那么多个页面实例之间又如何实现配置共享呢？

这个问题问的好，要处理共享问题，就可以利用函数将要共享的全局配置项给包装起来即可。

const createMyApp = options => { // 包装需要共享的全局配置项，并将相应的实例返回
  const app = createApp(options)
  app.component('my-component', {
    // ...
  })
  // 其他共享的功能
  return app
}

const foo = createMyApp({})
const bar = createMyApp({})
foo.mount('#foo')
bar.mount('#bar')

通过上面将需要共享的全局配置项包装起来后，那么foo和bar都将具有需要共享的全局配置项。

同时，全局API现已支持Tree Shaking。为何这么说？看个🌰：

// vue 2.x
import Vue from 'vue'
Vue.nextTick(() => {
  // ...
})

// vue 3.0
import { nextTick } from 'vue'
nextTick(() => {
  // ...
})

上述代码就是列举了 2.x 和 3 的nextTick写法，可以发现在vue 2.x中，nextTick是挂在Vue原型上的，那么打包工具在处理Tree Shaking时，即使在代码里并没有使用到nextTick，也依然会被打进包里面。恰恰相反的是，在vue 3.0中，已经支持nextTick从vue中按需引入，这样一来，就很好滴方便打包工具处理Tree shaking。

当然，在编写一个Vue插件时，我可以按下面写法：

// Vue 2.x
const plugin = {
  install: Vue => {
    Vue.nextTick(() => {
      // ...
    })
  }
}

// Vue 3.0
import { nextTick } from 'vue'
const plugin = {
  install: () => {
    nextTick(() => {
      // ...
    })
  }
}

好明显的是，上面代码会在打包时将Vue源代码也打进一个包中，无形之中就加大我们所编写的插件的大小。那有什么办法呢？

答案就是使用Webpack的externals属性，可有效减少插件的大小。

module.exports = {
  externals: {
    vue: 'Vue'
  }
}

# v-model

在Vue 2.x中，只能支持向子组件编写一个v-model，如下：

// 父组件
<my-component
	v-model:text="text"
	@update:text="text = $event"
></my-component>
// 简写如下
<my-component
	v-model:text.sync="text"
></my-component>
// 子组件
app.component('my-component', {
  template: `
		<input
			v-model="text"
			@input="$emit('update:text', $event.target.value)" />
	`,
  props: ['text']
})

而在Vue 3.0中，则支持绑定多个v-model在子组件中，并且可省略更新函数。

// 父组件
<my-component
	v-model:name="name"
	v-model:age="age"
></my-component>

// 子组件
app.component('my-component', {
  template: `
		<input
			type="text"
			v-model="name"
			@input="$emit('update:name', $event.target.value)" />
		<input
			type="text"
			v-model="age"
			@input="$emit('update:age', $event.target.value)" />
	`,
  props: ['name', 'age']
})

官方也推荐在Vue 2.x中使用sync修饰符全部修改为v-model形式。

另外，**Vue 3.0支持v-model实现自定义的修饰符——使用capitalize。**看个🌰：

<div id="app">
	<my-component v-model.capitalize="text"></my-component>
</div>

app.component('my-component', {
  props: ['modelValue', 'modelModifers'],
  template: `
		<input
			type="text"
			:value="modelValue"
			@input="emitValue"
		/>
	`,
  methods: {
    emitValue(e) {
      let { value } = e.target
      if (this.modelModifiers.capitalize) { // 判断是否含有自定义修饰符
        value = value.charAt(0).toUpperCase() + value.slice(1)
      }
      this.$emit('update:modelValue', value)
    }
  }
})

使用v-model时默认值都会使用modelValue，同时自定义修饰符使用的则是modelModifiers。

如果遇到一种情况是带有名称的v-model时，该如何处理？

这时，在子组件中依然使用该名称访问，但是自定义修饰符则需要使用名称 + Modifiers。

<div id="app">
	<my-component v-model:text.capitalize="text"></my-component>
</div>

app.component('my-component', {
  props: ['text', 'textModifers'],
  // ...其他处理逻辑
})

# v-if 优先级高于 v-for

在Vue 2.x中，如果在同一个元素上均使用v-if 和v-for时，那么会优先执行v-for，接着才会执行v-if。

在Vue 3.0后，v-if的优先级始终都会高于v-for。

# v-bind 绑定属性按顺序进行合并

一个元素上，如果v-bind="object"在同一个元素中同时使用语法和单个属性，这时候就会存在一个合并优先级问题。

先来看看Vue 2.x的合并，不管v-bind的位置在前还是在后，都会被原来的定义好的属性给覆盖。

<div id="foo" v-bind="{ id: 'bar' }"></div>
<!-- 解析后将得到 -->
<div id="foo"></div>

但是在Vue 3.0之后，是否覆盖则由编写v-bind=object的位置所决定。

<div id="foo" v-bind="{ id: 'bar' }"></div>
<!-- 解析后将得到 -->
<div id="bar"></div>

<div v-bind="{ id: 'bar' }" id="foo"></div>
<!-- 解析后得到 -->
<div id="foo"></div>

# 采用 defineAsyncComponent 定义异步组件

要引入一个异步组件，在Vue 2.x常常以下面这样的写法：

const asyncPage = () => import('./page.vue')
// 或者
const asyncPage = {
  component: () => import('./page.vue'),
  delay: 200,
  timeout: 3000,
  error: () => import('./errorComponent.vue'),
  loading: () => import('./loading.vue')
}

那么在Vue 3.0中，就可以直接使用defineAsynComponent定义异步组件。

import { defineAsyncComponent } from 'vue'

const asyncPage = defineAsyncComponent(() => import('./page.vue'))
// 或者
const asyncPage = defineAsyncComponent({
  loader: () => import('./page.vue'),
  delay: 200,
  timeout: 3000,
  error: () => import('./errorComponent.vue'),
  loading: () => import('./loading.vue')
})

可以看到，在Vue 3.0中将使用loader取替Vue 2.x中的component 。

# 生命周期名字更改

  • destoryed 更改为 unmounted；
  • beforeDestory 更改为 beforeUnmount；
