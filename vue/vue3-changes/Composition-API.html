# Composition API

如果您之前有了解过React hook，那么今天要总结的Composition API其实跟它就很类似，但也有区别（具体的区别有时间我会单独总结出来，当然您也可以上网搜搜来了解下🤔）。

那么，Composition API是什么？顾名思义，就是一个组合 API，能有效滴处理代码重用以及代码共享。

在大型复杂的应用中，Composition API所提供的代码重用和代码共享，恰恰能有效改善组件间维护问题，当然，Composition API推崇的则是Functional API ，可有效滴融入TS的存在。

当然，Vue 3.0为了兼容Vue 2.x，特地将继续支持原来那套写法，但是新版的写法会直接建议采用Compoition API。

接下来，就来简单地总结下在Vue 3.0中， Composition API都有哪些特性。

# setup

setup是针对vue 3.0中对组件所添加的新选项，其中触发的时机是，在Props解析后和组件创建之前。同样地，setup也是composition API的入口。

在执行的过程中，会先解析setup中代码，然后再解析外部Vue 2.x代码。

setup函数接受两个参数，分别是：

  • props：父组件传递下来的参数。
  • context：从组件实例中选择性地暴露一些propperty，如attrs、emit等。

举个🌰：

export default {
  name: 'my-component',
  setup(props, context) {
  	console.log(this) // undefined
    return {
      ...props
    }
  }
}

可以看到，由于setup是在组件创建之前才会触发的，那么在setup函数中就没法直接访问组件实例了，因此this为undefined。

所以，在setup 内部是没法直接访问到组件内定义的data、computed、methods。

# reactive

使用reactive可将一个引用类型作为响应式类型。（为什么我要侧重标明是引用类型？下面会提及）

如果您使用过Vue 2.x中Vue.observable()方法，那么reactive 是跟他等同的，直接点就是：reactive是 Vue.observable的重命名而已。

那么既然有了Vue.observable，为啥还要定义一个新的reactive？

引用官方的说法，也就是为了避免Rxjs中observable的命名冲突问题。

看个🌰：

<div @click="handleA">click</div>
<span>{{ a }}</span>

import { reactive } from 'vue'
export default {
  create() {
    console.log(this.state) // { a: 1 }
  },
  setup(props) {
    const state = reactive({
   		a: 1
    })
    const handleA = () => state.a++
    return {
      state,
      handleA
    }
  }
}

可见，使用reactive设置的响应式类型最终都是会挂载在组件实例上。

# ref

上面提及到，使用reactive时，只能将引用类型设置成响应式类型，那恰恰相反的是，ref特性则是用于将原始数据属性设置成响应式类型。看个🌰：

<div @click="handleStr">click</div>
<span>{{ str }}</span>

export default {
  setup() {
    const str = ref('haha')
    console.log(str.value)
    const handleStr = () => {
      str.value += 'hehe'
    }
    return {
      str,
      handleStr
    }
  }
}

虽然注意的是，在模板template中使用ref，是不需要写上.value的，同样地，如果模板使用的是reactive，则需要用上[object].xxx形式。

另外，如果将ref作为reactive对象的property时，会被解套value值。

import { ref, reactive } from 'vue'
export default {
  const str = ref('hehe')
  const state = reactive({
    str
  })
  console.log(state.str) // 'hehe'
	state.str = 'haha'
	console.log(str.value) // 'haha'
}

一定要注意的是，必须是作为reactive中对象的属性时才会被解套！！

那么，一旦作为reactive中不是对象的属性时，那么就需要自行访问其value属性。

// 放在数组中
import { ref, reactive } from 'vue'
export default {
  const str = ref('hehe')
  const state = reactive([str])
  console.log(state[0].value) // 'hehe'
	state[0].value = 'haha'
	console.log(str.value) // 'haha'
}

// 放在Map中
import { ref, reactive } from 'vue'
export default {
  const str = ref('hehe')
  const state = reactive(new Map([['str', str]]))
  console.log(state.get('str').value) // 'hehe'
	state.set('ref', 'gaga')
	console.log(str.value) // 'hehe'
}

在我个人看来，使用ref其实就是将该属性转化为使用{}来包装，这样一来就是指向一个地址了。

# toRef

针对父组件传递到子组件的props，我们都知道它是一个响应式的属性。

现在先来看个栗子，我们将子组件编写如下：

<span>{{ text }}</span>

export default {
  props: ['text'],
  setup(props) {
    const { text } = props
    return {
      text
    }
  }
}

当父组件中将text值修改时，正常情况下，子组件同样也会进行更新的。但是我可以肯定的是，上面解构代码会直接破坏响应式传递。

因此，toRef就是专门用来保持props响应式传递的问题的。

<span>{{ text }}</span>

import { toRef } from 'vue'
export default {
  props: ['text'],
  setup(props) {
    const { text } = toRef(props)
    return {
      text
    }
  }
}

经过上述的修改，当父组件更改text值时，子组件同样也会更新。

当然，如果父组件传递下来的值是reactive类型，那么就需要使用toRefs了，接下来就看个🌰：

# methods

看了上面的介绍后，相信您也知道可以直接在setup中定义方法。

export default {
  setup() {
    const handleStr = () => { // 处理字符串方法
    	// ...
    }
    return {
      handleStr
    }
  }
}

在setup的return中返回定义的方法，那么在模板中就可以直接使用。

# computed

在setup中同样也支持使用计算属性。举个🌰：

<input v-model="lastName" />

import { ref, computed } from 'vue'
export default {
  setup() {
    const firstName = ref('haha')
    const lastName = ref('hihi')
    const fullName = computed(() => `${firstName.value} ${lastName.value}`)
    return {
      lastName,
      fullName
    }
  }
}

另外，也可以直接提供get和`set方法。

export default {
  setup() {
  	const firstName = ref('haha')
    const lastName = ref('hehe')
    const fullName = computed({
    	get: () => {
        return `${firstName.value} ${lastName.value}`
      },
      set: (val) => {
        lastName.value += val
      }
    })
    fullName.value = 'hihi'
    console.log(fullName.value)
  }
}

可以看到的是，在setup中使用计算属性时，同样也是使用其value值。

# Lifecycle

在setup中使用生命周期函数和Vue 2.x中编写的会有些区别，主要的区别就在于，需要我们手动从vue中引入onXXX相应的生命周期。

对比Vue 2.x如下：

Vue 2.x Vue 3.0
beforeCreate、create setup
beforeMount onBeforeMount
mounted onMounted
beforeUpdated onBeforeUpdate
updated onUpdated
beforeDestory onBeforeUnmount
destoryed onUnmounted
errorCaptured onErrorCaptured
onRenderTracked
onRenderTriggered

onRenderTracked和onRenderTriggered都是用于调试的生命周期，其中onRenderTracked就是用于监测组件变化的（相当于一个追踪器），而onRenderTriggered则是用于监视触发组件变化的。

# watch

和Vue 2.x用法一样，用于监测响应式属性的变化，而在Vue 3.0的写法上则会有些不太一样。

watch接受三个参数：

  • 要监测的响应式属性。
  • 监测变化后的回调函数。
  • 额外的配置项（如deep、immediate选项）。
import { watch, ref } from 'vue'
export default {
  setup() {
    const str = ref('haha')
    const handleStr = () => { str.value = 'hihi' }
    watch(str, (newVal, oldVal) => {
      console.log(newVal, oldVal)
    })
    return {
      str,
      handleStr
    }
  }
}

当然，watch还能监测多个响应式属性。

import { watch, ref, reactive } from 'vue'
export default {
  setup() {
    const str = ref('haha')
    const state = reactive({
      a: 1
    })
    const handleStr = () => { str.value = 'hihi' }
    watch([str, state.a], ([newStr, newA], [oldStr, oldA]) => {
      console.log(newStr, newA)
    })
    return {
      str,
      state,
      handleStr
    }
  }
}

其实watch就是懒加载监听方式，可以加入immediate来立即执行，但它的终极目标则是更好滴去追踪变化属性。

# watchEffect

watchEffect也是一个用于追踪响应式变化，需要传入一个函数，当函数中所使用到的响应式属性变化时会直接触发该函数的执行。

watchEffect与watch 最大的区别就在于，watchEffect会在组件挂载后自动执行一次（相当于watch基础上加入了配置项immediate）。另外，watchEffect只要回调函数中使用到的任意一个响应式属性发生变化时，都会触发该回调函数执行，隐形之中也会产生一些副作用出来。

<div>
	<span @click="handleStr">Change the ref</span>
	<span @click="handleState">Change the state</span>
</div>

import { ref, reactive, watchEffect } from 'vue'
export default {
  setup() {
    const str = ref('haha')
    const state = reactive({
      a: 1
    })
    const handleStr = () => { str.value = 'hihi' }
    const handleState = () => { state.a++ }
    watchEffect(() => {
      console.log(str.value, state.a)
    })
    return {
      handleStr,
      handleState
    }
  }
}

上面的代码执行后，当随意点击任意一个点击函数后，都会直接触发到watchEffect中函数的执行（因为在watchEffect的处理函数中使用到了str和state.a两个响应式属性）。

也许您也看到了，由于任意一个响应式发生变化，都会触发watchEffect中函数执行，这必然会产生一些不必要的副作用，比如根据某一个响应式属性请求响应一些信息，由于额外的响应式属性发生变化，而触发多一次获取信息动作，那么此时上一次的请求咋办，最妥善的处理就是把上一次的请求删掉，避免副作用。

为此，watchEffect的回调函数接受一个onInvalidate函数参数，专门用于处理副作用的。

那么，onInvalidate函数在什么时候会触发呢，官方也给出了以下触发时机：

  • 副作用即将重新执行时。
  • 侦听器被停止。

下面直接举个简单🌰：

import { watchEffect } from 'vue'
export default {
  setup() {
    const name = ref('Tom')
    const age = ref(0)
    watchEffect(async (onInvalidate) => {
      onInvalidate(() => {
        message.cancel() // 取消上一次的请求
      })
      const message = await getUserInfo(name.value)
      age.value = message.age
    })
  }
}

当然，我们还能处理一些计时器，当侦听器中使用了计时器，在被停止之际需要清理掉计时器，避免内存泄漏。

# provide/inject

provide/inject 就等同于react 中的context，都是为了处理两个通信组件之间相隔多个组件时的情形。

接下来，我们就来看看在Vue 3.0中是如何使用provide/inject的。

// 父组件
import { provide } from 'vue'
export default {
  setup() {
    const str = ref('haha')
    const state = reactive({
      a: 1,
      b: 2
    })
    provide('str', str)
    provide('state', state)
  }
}

// 孙子组件
import { inject } from 'vue'
export default {
  setup() {
    const str = inject('str')
    const state = inject('state')
    return {
      str,
      state
    }
  }
}

现在问题来了，由于父组件传递下来的数据，若孙子组件想改动，如何处理？

相信这种场景还是很常见的，只需紧记Vue通信的核心就是：Props down, events up。为此，我们同样也可以通过在provide中传递events，然后孙子组件就可以inject来反向调用该方法以及对数据进行改动。

// 父组件
import { provide } from 'vue'
export default {
  setup() {
    const str = ref('haha')
    const state = reactive({
      a: 1,
      b: 2
    })
    const updateState = () => { state.a++ } // 定义更新响应式属性的方法
    provide('str', str)
    provide('state', state)
    provide('updateState', updateState)
  }
}

// 孙子组件
import { inject } from 'vue'
export default {
  setup() {
    const str = inject('str')
    const state = inject('state')
    const updateState = inject('updateState')
    return {
      str,
      state,
      updateState
    }
  }
}

# 代码组合以及复用

上面我提及过，Vue 3.0所提倡的Composition API，是致力于提高代码的逻辑组合以及进一步对代码进行复用。

Composition API的相关特性在上面基本都有了一个大概的讲解，但是还没真正地体现到Composition API的核心价值，为此我们通过官方给出的栗子🌰来看看，相信你睇完后，肯定对Composition API有进一步的了解～（😂我猜的）

先看下面使用Vue 2.x进行开发的组件代码。

export default {
  props: ['class'], // 班级【1】
  data() {
    return {
      students: [], // 学生【1】
      score: 0, // 分数【2】
    }
  },
  computed: {
    studentsMatchScore() { // 根据分数对学生进行筛选【2】
     	return this.students.filter(item => item.score >= this.score)
    }
  },
  watch: {
    'class': 'getStudents' // 监听班级的变化来获取新的学生【1】
  },
  methods: {
    async getStudents() { // 根据班级来获取学生【1】
      // ...
    }
  },
  mounted() {
    this.getStudents() // 【1】
  }
}

上述的代码，如果单纯从Vue 2.x的角度来看的话，真的觉得很普通，并没有啥问题。

但是当你细心检查，就会发现代码参差不齐，啥意思？

说白了，就是处理同一件事的代码并没有聚集在一个地方进行处理，在上面我已经表明符号，其中【1】表示获取学生处理逻辑，【2】则是对获取到的学生进行分数筛选逻辑，好明显，最理想的状态就是能够独自封装它们，这样对于日后的维护将会得到一个质的提升。

那么，Vue 3.0就是考虑到这一点，才会提出Composition API （当然Composition API远不止为了这件事😂）

下面就使用Composition API对上述的代码进行优化。

// useGetStudents.js
import { ref, watch } from 'vue' 
export default function useGetStudents(class) {
  const students = ref([]) // 学生
  const getStudents = () => { ... } // 获取学生
  onMounted(() => {
    getStudents() // 在组件挂在后就直接获取学生
  })
  watch(class, (newClass, oldClass) => getStudents) // 根据班级的变化获取相应的学生
  return {
    students,
    getStudents
  }
}
  
// useStudentsClassSearch.js
import { computed } from 'vue'
export default function useStudentsClassSearch(students) {
	const score = ref(0)
  const studentsMatchScore = computed(() => students.filter(item => item.score >= score))
  return {
    score,
    studentsMatchScore
  }
}
  
// myComponets.vue
import { toRefs } from 'vue'
import useGetStudents from './useGetStudents.js'
import useStudentsClassSearch from './useStudentsClassSearch.js'
export default {
  setup(props) {
    const { class } = toRefs(props)
    const { students, getStudents } = useGetStudents
    const { score, studentsMatchScore } = useStudentsClassSearch
    return {
      students,
      getStudents,
      studentsMatchScore
    }
  }
}

睇完上面优化后的代码，是否有点阔然开朗？

是的，将【1】和【2】各自代码都独自封装起来，这样一来，就可以很好滴管控各自单独的逻辑代码，因为他们中每一份封装都是只做一件事情。

对于封装好的逻辑代码，就可以选择性地组合起来，当然也可以继续进行复用，也刚好印证了Composition API的大核心--组合以及重用。
