# 脚手架那回事

如果你平时使用过vue-cli或者create-react-app，那么“脚手架”这个词，你肯定不会陌生。下面就来谈谈脚手架到底是咋回事。 相应的Github仓库在此https://github.com/Andraw-lin/c5-cli，喜欢的可以丢一个star🐶。

# 遇到的问题

首先，我必须承认，vue-cli以及create-react-app已经是很成熟乃至很优秀的脚手架，直接使用他们基本是没啥问题。但是，当你在一个公司开发h5时候，难免会出现以下问题。

  • 由于日常处理逻辑基本一致（如分享、axios封装、方向代理、优化等等），必须在vue-cli或create-react-app基础上再抽离一层共用层，做成一个满足日常开发需要的业务架构。
  • 抽离出来的业务架构，在每次新的h5活动开发中，都需要手动复制粘贴文件来进行开发。

当遇到上述问题，你是否会像我当时一样觉得很苦恼😵。。一旦静下心来，你是否会思考，能不能自己做一个符合日常开发需要的脚手架呀？如果你有这样的想法，那就妥妥的了（至少证明你是个会思考的人😄）。下面我会以一个简版的脚手架开发进行简单介绍，毕竟网上太多的脚手架教程，我觉得有时候教程多并不是一件很好的事情，毕竟一定要别人看懂才是最好的。因此下面我会使用一个简版脚手架来分析，希望能帮到大家啦～

# 前提条件

在真正开发一个脚手架前，你必须先清楚以下几个NPM包的用意。

  1. commander：作为tj的一个神作（github地址在此），用于定义nodejs相应的命令（毕竟它的用途就是让用户更友好地适应nodejs命令行界面）。
  2. fliglet：简单来讲，就是为了输出特别好看的图案，图案是经过一个字符一个字符地来拼凑出来的（NPM地址在此）。
  3. download-git-repo：用于下载相应的github或者gitlab开发模板（NPM地址在此）。
  4. handlebars：动态解析模板引擎，在这里主要是根据用户输入的内容动态更新package.json（github地址在此）。
  5. inquirer：基于Node的与用户进行交互的命令行工具，相当于机器问你，用户负责回答，然后保存回答的内容（简单来说，就是一问一答，然后保存起来）（github地址在此）。
  6. ora：用于命令行loading的加载图标展示（github地址在此）。
  7. chalk：修改命令行字体展示（github地址在此）。
  8. log-symbols：用于为各种日志提供着色的符号，如错误或正确的警示（github地址在此）。
  9. shelljs：用于动态执行命令，如js中的eval方法（github地址在此）。

如果你对上述都有所了解过，那可以直接到下一个话题，当然若你都没了解过的话，我还是建议你先了解一下比较好一些，免得到后面不知道这个是干啥用的～

# 初始化

先找到一块空地方，然后呢，执行如下命令：

npm init -y

-y参数用于使package.json中所有参数都采取默认形式，当然如果你想填好每一项（如version、name、author等等），可以直接去掉该参数。init后的文档结构是这样的。

|- package.json

继续执行如下命令，创建bin目录以及在bin目录下创建init.js。

mkdir bin
cd bin
touch init.js

执行完所有的命令后，会得到下面这个目录结构。

|- bin
|- - init.js
|- package.json

入口文件就在init.js，我们接下来就会在该文件上进行相应的开发。

#!/usr/bin/env node
// 用于告诉系统使用/usr/bin下的node来解释当前文件

const program = require('commander'); // 用于动态识别用户输入命令
const figlet = require('figlet'); // 用于展示好看的输出图案

program
  .version('1.0.0', '-v, --version')
  .command('create <type> [name] [otherParams]') // 定义用户创建项目的命令
  .alias('cli')
  .description('Generates new code')
  .action(function (type, name, otherParams) {
    figlet.text('C5-CLI', {
      horizontalLayout: 'fitted'
    }, (err, data) => {
      console.log(data); // 展示输出图案
      switch(type) {
        case 'download':
          const downloadFunc = require('./download.js');
          downloadFunc(name);
          break;
        default: return false;
      }
    })
  });

program.parse(process.argv);

在上述的代码中，我尤其需要注意的是#!/usr/bin/env node，它的意思就是使用/usr/bin下的node来解析当前入口文件的。接下来就是开始使用commander实例来定义相应的指令，以及版本号。

代码看起来难度并不大，从官方相应的API就可以很好滴参考。

# 加载模板

接下来就到了我们的重头戏了，有了规范定义好的指令后，想必你肯定知道下一步就是要直接从github以及gitlab上下载相应的模板代码下来啦。那么问题来了，怎么去下载呢？

想必你们从前面知道了答案，那就是直接使用download-git-repo库进行下载。我们就直接看代码吧，比较直观一些～

首先，执行如下命令。

cd /bin && touch download.js

然后，可以直接观看一下download.js里面到底是怎么编写的～

const fs = require("fs");
const download = require("download-git-repo"); // 用于下载相应的github或者gitlab开发模板
const handlebars = require("handlebars"); // 根据用户输入的内容动态更新package.json
const inquirer = require("inquirer"); // 基于Node的与用户进行交互的命令行工具
const ora = require("ora"); // 用于命令行loading的加载图标展示
const chalk = require("chalk"); // 用于修改命令行字体样式
const symbols = require("log-symbols"); // 为各种日志级别提供着色的符号--错误正确警示
const shell = require('shelljs'); // 用于自动执行一些自动化命令

const downloadFunc = async (name) => {
  if (!fs.existsSync(name)) { // 判断是否已经存在相应的项目名
    const answers = await inquirer.prompt([
      {
        name: "description",
        message: `请输入项目描述`,
      },
      {
        name: 'version',
        message: '项目的版本号',
        default: '1.0.0'
      }
    ]) // 用户交互的相应问题
    const spinner = ora({
      text: "正在下载模板...",
      spinner: "bouncingBar"
    });
    spinner.start(); // 加载展示相应动画与文案
    download("direct:https://github.com/***/***.git#***", name, { clone: true }, err => { // 拉取相应的github模板，使用https时必须使用direct关键字
      if (err) {
        spinner.fail();
        console.log(symbols.error, chalk.red(err));
      } else {
        console.log(symbols.success, chalk.green('项目初始化完成'))
        const fileName = `${name}/package.json`;
        const meta = {
          description: answers.description,
          version: answers.version
        };
        if (fs.existsSync(fileName)) { // 在现有的package.json文件中进行补充动态内容
          const content = fs.readFileSync(fileName).toString();
          const result = handlebars.compile(content)(meta);
          fs.writeFileSync(fileName, result);
        }
        const installSpinner = ora({
          text: chalk.yellow('正在安装依赖...')
        })
        installSpinner.start()
        shell.cd(name)
        const installObj = shell.exec('cnpm install') // 自动执行cnpm install
        if (installObj.code !== 0) {
          console.log(symbols.warning, chalk.yellow('自动安装失败，请手动安装！'));
          installSpinner.fail(); // 安装失败
          shell.exit(1);
        }
        installSpinner.succeed();
        console.log(symbols.success, chalk.yellow('安装依赖成功...'))
      }
    })
  } else { // 项目已存在，不再处理
    console.log(symbols.error, chalk.red("项目已存在"));
  }
};

module.exports = downloadFunc;

针对上述的代码，有几个点是需要注意的。

  • 使用download-git-repo库在填写相应的github或gitlab仓库时，若使用的是HTTPS，那么必须要在前面加上direct:以及在后面加上#分支名。否则很容易就会报128状态码。
  • 在使用自动执行脚本shelljs库时，必须要手动进入下载好的模板shell.cd(目录名)，然后再执行，否则就无法进行相应的依赖安装。

接下来我们就来简单滴分析上述的代码。

  • inquirer用于与用户在命令行界面上进行相应“你问我答”，异步的，可有效滴结合async/await。
  • 使用handlebars可将inquirer中所存储的答案进行扩展到一些需要拓展文件，如package.json。

至此，整个简易版的脚手架开发就到此结束，看起来应该不难吧🤦‍♂️，虽然我一开始看到网上关于脚手架的淋漓满目文章介绍时，都是懵逼的（因为很多都是一带而过，当然可能也是我跟不上别人节奏吧😂）。
